Asset pipeline · Research note · 2026-05-19

Paid 3D backends

What it would take to add commercial 3D-generation services to the existing pipeline as opt-in backends, with quality, licensing, and cost analysis grounded in current (May 2026) pricing.

Key takeaway

Paid services largely solve two problems the local Apple-Silicon pipeline has — the missing CUDA rasterizers that gate Hunyuan/Rodin PBR, and the licence ambiguity around community-licensed models. At realistic usage (~20–200 generations per month) a pay-per-call account on fal.ai or a single Tripo Pro subscription runs $1–35/month total. Add them as opt-in generator strings to generate.sh the same way SF3D and SPAR3D work today; the default stays Z-Image Turbo → SF3D → Blender → Snapmaker U1.

01 · Context

Why consider paid backends at all

The local pipeline is great for the default path. Three forces push toward paid as soon as you want maximum quality:

  1. The Apple-Silicon CUDA gap. The shape side of every modern model runs fine on MPS. The texture side — PBR map synthesis, differentiable rasterizers — is almost universally CUDA-only today. On a Mac you either get untextured shape (and lose half the quality story) or you wait for community forks that lag months behind upstream. Paid cloud services run on CUDA GPUs and don't have this problem.
  2. The licence gap. Hunyuan3D-2 and Hunyuan3D-Paint ship under the Tencent Hunyuan Community License (revenue thresholds, region exclusions). TRELLIS is non-commercial. Paid services with explicit "you own all assets, full rights to distribute and sell" terms are the clean path to commercial usability for Grithkin / GripCraft / 3D-print products.
  3. Time vs money. Two Mac Studios at ~$10,000+ in hardware. Cloud-quality output at ~$0.05–0.50 per generation. The hardware doesn't beat the cloud until you're running ~20,000 generations a year — past which Tripo or fal.ai pay-per-call still wins by 5–20×.
Context — the architecture already supports this

A paid backend is just another value for generate.sh -g GENERATOR. The wrapper would POST to an HTTP API, poll until done, download the GLB, then drop into the same Blender cleanup as SF3D. Licence-bucket map gets a new entry per backend. Manifest + benchmark output records the backend choice. No core change.

Rationale — why fal.ai keeps coming up

It's the cleanest pay-per-call host for multiple model families at once: same API shape across Tripo, Rodin, Hunyuan, TripoSR. A single FAL_API_KEY in ~/3d-pipeline/.config unlocks every backend the pipeline ever wants to add, vs. signing up for and managing four separate subscriptions. The markup over running on a rented GPU is modest and pays for the integration work you'd otherwise do yourself.

Decision — paid services pay for themselves quickly

If you would otherwise re-generate an asset 5 times on Apple Silicon trying to get an acceptable result, one paid generation is cheaper than your time. The cheap-vs-fast tradeoff inverts the first time your time-cost crosses ~$5/hour.

06 · Architecture

How a paid backend fits the v0.2 pipeline

Structurally identical to how sf3d, spar3d, and trellis work today. The only new shape is "HTTP API call" replacing "local venv invocation." Blender cleanup, manifest update, engine staging, the queue, and the benchmark harness all keep working because they only care about the GLB on disk.

The work per backend

  1. Add to license_bucket_for_model in scripts/_pipeline_lib.sh — one new case per backend.
  2. Add an elif branch to scripts/generate.sh that calls the new paid_backend.py helper with the right API endpoint and parameters.
  3. Add the API key to ~/3d-pipeline/.config (e.g. fal_api_key = ...), read via the existing read_pipeline_config helper. Wrappers fail with clear install guidance if the key is missing — same pattern as SPAR3D's directory check.
  4. Add SKILL.md routing so Claude Code can suggest the right backend per task class.
  5. Embeds + parity + tests via the existing make regenerate / make verify pipeline. No new tooling.

Estimated effort: ~2–4 hours per backend for the first one, ~1–2 hours each thereafter (shared HTTP plumbing).

07 · Decisions

Recommendations · by use case

Grithkin / GripCraft game assets

→ Tripo Studio

Clean quad topology, ~10s/gen, commercial_safe on paid plan. $0.05/gen pay-per-call or $19.90/mo subscription for 3,000 credits.

3D-printable hero pieces

→ Hunyuan3D 2.x (post-review)

Top geometry detail and full PBR via cloud. Same Tencent licence as --mode paint — gate until reviewed. $0.48/gen textured.

Display / portfolio / hero renders

→ Hyper3D Rodin

4K PBR, photorealistic, highest-end output. $0.40/gen via fal.ai — perfect for rare assets that have to look perfect.

Throwaway proxy / draft geometry

→ Local SF3D (no change)

Free, fast enough, already installed. Don't pay for drafts that you'll regenerate.

Decision — the default stays the default

No proposal here changes the default lane (Z-Image Turbo → SF3D → Blender → Snapmaker U1). Paid backends are opt-in generators selected with -g tripo-fal or -g rodin-fal. The Claude Code skill picks the right backend based on the task class and your hardware_tier, but defaults to the local commercial-safe path when in doubt.

10 · Decisions

Implementation plan · proposed sequence

Three small PRs, mirroring the per-phase cadence of the v0.2 work. Each one ships independently behind the wrapper's existing -g GENERATOR switch — no default change at any step.

  1. PR 1 — tripo-fal as the primary game-asset backend. Adds scripts/paid_backend.py (shared HTTP helper for fal.ai calls), one new branch in generate.sh, license-bucket entry, SKILL.md routing, and a smoke test through the existing --json + benchmark plumbing. Largest first because the shared HTTP helper lands here.
  2. PR 2 — rodin-fal as the hero backend. ~30 minutes once PR 1 is in. Reuses the HTTP helper; adds the licence-bucket entry, the generate.sh branch, and a SKILL.md note that it's the hero-asset choice.
  3. PR 3 — hunyuan3d-2-fal as a licence-gated lane. Mirrors the existing --mode paint placeholder pattern: accept the option but refuse to run with error=needs_license_review until the Tencent licence has been reviewed against commercial use. Removes the gate by deleting one block once review is complete.

Optional: PR 4 — Meshy or TripoSR as additional opt-in backends if usage data from the first three PRs suggests value. Skip for now.

Rationale — start small, prove the shared plumbing

PR 1 is the one with risk: it introduces paid_backend.py and decides on the ~/3d-pipeline/.config key convention for API keys. PRs 2 and 3 are mechanical once PR 1 lands. If something feels wrong about the shared helper, we want to learn that before the second backend doubles down on the same design.

11 · Working with this

Open questions for Ken

  1. fal.ai account? The cheapest credible path to actually using any of these. ~$10 starter credit covers months of prototyping. Single API key in ~/3d-pipeline/.config unlocks every backend the pipeline ever wants to add.
  2. Subscription vs pay-per-call for Tripo? The breakeven is ~400 generations/month. Below that, fal.ai pay-per-call wins; above, the $19.90 Tripo Pro subscription wins. You'll need real usage data to choose — start with pay-per-call.
  3. Hunyuan3D licence review timing? Same blocker as the existing --mode paint placeholder. PR 3 ships the gate; only PR-4-equivalent (lift the gate) needs the review.
  4. Single PR or three? Three small PRs are reviewable and roll back independently. One big PR is faster to land. Same cadence question as v0.2.
